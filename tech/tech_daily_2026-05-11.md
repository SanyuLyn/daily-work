# 技术日报 2026-05-11

## Flutter

### [Flutter Web 正式移除 HTML renderer，只支持 CanvasKit 和 SkWasm](https://zhuanlan.zhihu.com/p/11809045697)
**摘要**：Flutter 官方宣布彻底弃用 HTML 渲染器，该特性已正式合并至 beta 版本。此后 `--web-renderer` 参数将不再生效，Flutter Web 将仅支持 CanvasKit 和 SkWasm 两种渲染方案。CanvasKit 借助 WebAssembly 运行 Skia，渲染效果接近原生；SkWasm 是性能更优的下一代选项。开发者需尽快评估应用在两种模式下的兼容性，重点关注字体加载与文本排版的差异点，并完成迁移测试。

### [Flutter 2026 Roadmap 发布，未来计划是什么？](https://zhuanlan.zhihu.com/p/2009657387510952563)
**摘要**：Flutter 团队发布2026年官方路线图，核心目标包括：完成 Android 平台 Impeller 渲染器迁移并移除旧版 Skia 后端（针对 Android 10+）、实现 Android 17 与新版 iOS 的首日支持、持续改善 Web 端的可访问性及多窗口桌面环境体验。Dart 语言层面，2026年计划引入 Primary Constructor 简化类声明，以及 Augmentations 语法糖以简化代码生成场景。同时将提供新版 Material 3 Expressive 和 Liquid Glass 样式作为可选 UI 包。

### [新年 Flutter 3.41 放大招，历经3年，多窗口支持终合并](https://zhuanlan.zhihu.com/p/2005389425048307284)
**摘要**：Flutter 3.41 正式合并了多窗口桌面支持，这一特性历经三年开发，背后主要推动力来自 Canonical（Ubuntu 背后的公司）。多窗口支持让 Flutter 桌面应用可以真正创建和管理多个原生窗口，是 Flutter 冲击桌面生产力工具场景的关键一步。相比 Compose Multiplatform 更早落地的多窗口能力，Flutter 的延迟根源在于渲染引擎与平台窗口系统解耦架构带来的跨窗口状态共享复杂性，此次合并标志着这一难题得到了工程化解决。

### [Flutter 鸿蒙 2026 路线发布，加速同步官方生态，进一步优化体验](https://zhuanlan.zhihu.com/p/2016571565735682780)
**摘要**：Flutter-HarmonyOS 社区发布2026年路线图，核心目标是将与官方 Flutter 版本的适配周期从7个月以上压缩至4个月以内，实现更快速的生态同步。路线图还计划进一步优化鸿蒙平台上的渲染性能、改善与 ArkTS 组件的互操作体验，并系统梳理三方库适配状态。对于在鸿蒙生态布局的移动开发团队，这份路线图是评估技术路线的重要参考。

### [2026年 Web/App 跨端开发方案终极对比：React Native vs Flutter vs Uni-app vs Taro（9年实战选型指南）](https://zhuanlan.zhihu.com/p/2010120152784585953)
**摘要**：拥有9年实战经验的开发者对四大跨端方案进行深度横向对比。结论：追求极致性能与精美 UI（如动画密集型产品或游戏化界面）、或需同时覆盖移动端与桌面端的项目，Flutter 是最佳选择；Uni-app 和 Taro 更适合以微信小程序为核心的国内业务场景；React Native 生态成熟但性能上限不及 Flutter；纯 Web 技术栈在复杂动画场景下仍存在明显差距。文章附有详细的选型决策矩阵。

## Android

### [Android 17 Beta4 发布：四大行为变更，不改上线就崩](https://zhuanlan.zhihu.com/p/2029480668401349168)
**摘要**：Android 17 Beta 4 正式发布，被官方定义为"关键里程碑"版本，测试环境已接近最终形态。四大重点行为变更：① 基于设备 RAM 的应用内存上限强制执行；② 引入设备侧异常检测服务（结合 ProfilingManager）；③ 面向 Android 17+ 目标版本的应用，方向锁定与宽高比限制在大屏设备上将强制失效，应用须填满窗口；④ 高级防护模式（APM）正式落地，开启后将限制侧载、USB 数据传输及高风险权限行为。官方建议开发者立即针对 Beta 4 开展回归测试。

### [Android 17 有什么需要适配的？2026 Android 禁止侧载又是什么？](https://zhuanlan.zhihu.com/p/2009937286335308608)
**摘要**：全面梳理 Android 17 的适配要点及2026年谷歌对侧载（Sideload）的管控新政策。谷歌正式推出"高级 Android 侧载流程"，包含六项复杂验证步骤和强制24小时等待期，大幅提高非 Play 应用的安装门槛。Android 17 适配重点包括：新版 Photo Picker 权限模型变更、后台任务进一步限制、以及未经开发者认证的应用将被拦截。开发者认证计划：2026年3月面向所有开发者开放，9月起在部分地区强制执行。

### [Google 发布 Android CLI：打造面向 Android 工程的 Agent 能力](https://zhuanlan.zhihu.com/p/2029480482035737404)
**摘要**：谷歌于2026年4月发布 Android CLI 工具集，旨在将 AI Agent 能力深度整合进 Android 开发工具链。该套件包含三大组件：Android CLI 命令行工具（支持自然语言指令完成编译、调试、模拟器管理等任务）、Android Skills 技能库（封装常见 Android 工程操作流程）、Android Knowledge Base 知识库（提供 Android 平台专项知识支撑）。这是谷歌将 AI 辅助开发能力从 Gemini Code Assist 延伸至 Android 生态的重要举措。

### [Google 开始正式强制 Android 适配 16K Page Size，你准备好了吗？](https://zhuanlan.zhihu.com/p/1904196757702834168)
**摘要**：谷歌已正式要求 Android 应用适配 16K 页面大小。自2025年11月1日起，所有提交至 Google Play 的新应用及更新包（面向 Android 15+ 设备）必须支持16KB页面大小。开发者需检查 native so 库的编译配置，使用正确的对齐参数（`--build-id` 及 `PT_LOAD` 段对齐）重新编译，并更新 CMakeLists.txt 或 Android.mk 中的链接器脚本。未适配的应用在使用16K页内核的设备上将直接崩溃，切勿忽视。

### [Android Gradle Plugin 9.0 发布，为什么这会是个史诗级大坑版本](https://zhuanlan.zhihu.com/p/1997655149036995133)
**摘要**：AGP 9.0 正式发布，作者将其定性为"史诗级大坑"。主要破坏性变更包括：移除旧版 Groovy DSL 的兼容支持（须迁移至 Kotlin DSL）、Java 17 成为编译最低要求、多个构建 API 被废弃且缺乏直接替代。大量依赖自定义 Gradle Task 或旧版第三方插件（如 Crashlytics、代码映射工具等）的项目升级后将面临构建失败，建议团队在独立分支充分验证后再推进升级。
